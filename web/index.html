Question 1: What is the output for following program (./file.elf) in a 64-bit machine ?

$ ls

file1.c  file2.c  file.elf

 

$ cat file1.c

void* fun(char *addr)

{

 return addr;

}

 

$ cat file2.c

main()

{

 char a;

 char *p = fun(&a);

 *p = 10;

 printf ("%p -> %d\n",p, a);

}

 

$ gcc -S file1.c file2.c -o file.elf

 

Answer:

 ./file.elf

Segmentation fault (core dumped)

Reason:

By default function return type is int (4 bytes). Since the address of 64-bit machine is 8 bytes. The MSB will not be returned from the fun(). Leads to invalid access by pointer p in main().

 

// *p = 10;

 printf ("%p -> %p\n",p, &a);

 

$ ./file.elf

0xfffffffff68d4b9f -> 0x7ffff68d4b9f

1. What are re-entrant and non-reentrant functions ? How they will impact in OS ?

Answer

 say you have a function.. when it is running, a higher priority interrupt takes the CPU context to another process.. the CPU will leave the flow of the current function as such. when it gets back to our process, if the CPU can "continue" your function flow, from where ever it left it.. it is called re-entrant function.

simple thing is.. if your function has only local variables, it can be re-entrant

say for example you have a function that has a pointer to a memory.

 

so, if a funtion uses some global buffer or shared data, then it is not a re-entrant function right ?

 

 when we pause the function, then continue later.. but in that time, the memory is changed. But the place where you "continue" from doesn't refer the memory again. but uses the old value itself. it'll become a problem. this would be an non-reentrant function.

when you are writing a code which can be interrupted by another process, for example in networking devices, better to have reentrancy as it is safer.

2. Hard link and Soft link in linux: (say we have two files file1 & file2).

Answer

        say we have two files file1 & file2 in “file1 file1_hard" will create a hard link to file1, that points to file's inode (file contents directly). So even if the actual file file1 is deleted, the hardlink will still work.

"ln file2 file2_soft" will create a soft link to file2, that points to file2's path. This is like a shorcut that we create in windows. If the actual file is renamed or deleted, the soft link file will point to non-existing location and will be broken.
